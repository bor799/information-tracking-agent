# AI 信息信号引擎

**从静默流水线到自进化 Agent 系统**

一个面向长期主题追踪、信号萃取和个性化知识沉淀的 AI 系统。

它不是简单地抓取文章、总结文章、推送文章。它持续观察用户关注什么，自动发现新信息，分层调用模型，暴露失败风险，并把高价值内容沉淀成可复用的知识信号。

它的目标不是"读更多文章"，而是从持续流入的信息中，提炼出真正值得用户决策和重读的 0.1% 信号。

---

## 为什么这个系统必须存在

这个项目最初不是失败于性能，而是失败于"看起来正常"。

系统每天定时抓取、筛选、分析、推送文章。日志显示流程完成，数据库也有新数据。但一次 LLM API 异常后，系统没有报错——旧逻辑为了"保证流程继续运行"，使用文章字数长短估算评分。于是 200 篇文章几乎全部被打成 7.0 分通过初筛，后续深度分析静默失败。主流程显示运行完成。

系统没有罢工。它只是悄悄产出了错误结果。

从那一刻开始，我们意识到：**AI 系统最危险的不是慢，也不是贵，而是失败时没人知道。**

不是我们想炫技重构，而是旧系统的失败模式已经不可接受。

---

## 三代进化：决策权如何迁移

这个系统经历了三代进化。每一代的核心区别不是功能多少，而是谁在做决策：

```
V1：程序在决策
    ↓
V2：Agent 开始参与决策
    ↓
V3：系统能根据反馈调整自己的决策方式
```

---

### V1：勤奋但愚笨的计算器

**旧问题**

V1 是一个典型的确定性流水线：Cron 定时启动，扫描最近 7 天 RSS，逐篇调用 LLM，根据评分进入下一阶段，最终推送结果。这套系统在传统后端场景里看起来合理，但它有一个致命假设——每一步失败都会显式报错。

**触发事故**

一次 LLM API 调用异常后，系统没有中断，也没有报警。旧逻辑使用文章字数长短估算评分，大量文章被错误打成 7.0 分通过初筛。后续深度分析阶段实际全部失败，但主流程显示运行完成。

**根因**

确定性流水线不适合 AI 的不确定性。系统假设"流程跑完"等于"结果正确"，把容错设计成了掩盖错误。

**设计原则**

> AI 系统不能把"流程跑完"当成成功。真正重要的是：每一个判断是否可信。

---

### V2：从脚本到 Agent 的转折

V2 的目标不是让脚本更复杂，而是减少脚本对 AI 的错误控制。我们保留确定性系统负责边界——调度、状态、失败记录、成本控制、数据落库。但把内容判断、语义筛选和复杂分析交给模型。

```
V1: Cron → 盲扫 → 规则判断 → LLM → 静默降级 → 推送
V2: 调度器 → 增量感知 → 轻量模型初筛 → 旗舰模型深挖 → 失败队列/告警 → 知识沉淀
```

#### 动作一：增量感知

旧系统每次醒来都盲扫最近 7 天。这带来两个问题：重复处理大量旧文章；`RSS_DAYS_LIMIT = 7` 这类硬编码无法适配不同调度频率。

V2 引入调度状态记忆。系统记录每个信源上一次成功读取的位置。如果存在历史状态，就只抓新增内容；如果没有历史状态，就根据调度间隔动态计算回滚窗口。调度器从"定时执行脚本"变成了"带着记忆醒来的 Agent"。

代码层面，`URLDeduper` 在调度启动时加载已有 URL 集合，`scheduler.py` 基于历史状态动态计算 lookback 窗口，`RSS_DAYS_LIMIT` 硬编码被彻底移除。

#### 动作二：认知资源分层

旧系统把所有任务都交给旗舰模型，既昂贵，也不符合人类处理信息的方式。

V2 把 AI 调用分成两层：轻量模型（`scoring_model`）负责粗筛，判断文章是否值得继续看；旗舰模型（`extraction_model`）负责深挖，分析结构、提炼观点、识别长期价值。这不是简单的成本优化，而是认知资源管理——不是所有信息都值得消耗最高级别的模型注意力。

#### 动作三：Fail Loudly

这是最重要的工程哲学。V2 删除了所有"假装成功"的降级逻辑。

如果 LLM API 失败：不再用字数估分，不再自动放行，不再写入伪结果。系统会立即记录失败原因，写入失败队列，触发告警，阻断污染数据进入知识库。

代码层面，`FailureKind` 枚举定义了 12 种类型化失败（`llm_rate_limit`、`llm_timeout`、`auth_invalid`、`fetch_failed` 等），队列状态机明确区分 `done`、`rejected`、`retry_scheduled`、`failed_terminal`——绝不把失败标记为成功。

**设计原则**

> 宁可让系统停下来，也不允许错误结果安静地流入下游。

---

### V3：具备持续进化能力的系统

V3 把用户反馈、阅读行为、历史萃取记录变成了系统进化的输入。核心不是"系统活了"，而是——系统会根据你如何使用信息，调整以后如何处理信息。

#### 记忆分层

V3 引入两层记忆。**短期焦点**记录用户最近 1-2 周密集关注的主题，用于动态提高相关内容的筛选权重。**核心画像**记录用户长期稳定的价值观、研究框架和判断偏好，用于影响长期信号萃取标准。短期焦点解决"最近在关心什么"，核心画像解决"长期真正看重什么"。

```
用户行为
  ├─ 最近搜索 / 最近重读 / 最近标记
  │      ↓
  │   短期焦点记忆 → 调整近期信号权重
  │
  └─ 长期偏好 / 投资框架 / 判断模型
         ↓
      核心画像记忆 → 影响长期萃取标准
```

#### 从配置注入到兴趣反推

旧系统依赖用户手动配置关注主题。V3 不只接受配置，还会根据历史行为反推关注方向。用户反复重读某类文章、频繁搜索某个主题、持续标记某类内容为高价值——系统会把这些行为视为隐性反馈，并提出新的关注方向建议。

#### 技能内生化

V2 的抓取能力依赖外部工具链。主进程像调度员，只能调用外部抓取器，再等待结果返回。V3 将抓取、解析、自检、异常恢复能力内生到运行时中。`multi_channel.py` 实现了 9 个平台适配器（YouTube、Twitter/X、Reddit、V2EX、Hacker News、微信公众号、小宇宙 FM、RSS/Atom、Web），每个适配器内置多级降级策略。

系统不再只是调度外部工具，而是能够感知工具链状态、识别抓取失败原因，并执行恢复策略。抓取失败不再只是一个空结果，反爬不再只是一个未知异常，工具链断裂不再需要人工第一时间介入。

**设计原则**

> V3 不再是"指挥机器人干活的包工头"。它不仅能处理信息，也能感知自己的工具是否失效，并尝试恢复。

## 我们真正解决的五类失败模式

| 失败模式 | 问题 | 解法 |
|----------|------|------|
| **静默失败** | LLM API 失败后，旧系统继续生成伪结果 | Fail Loudly：类型化失败队列，告警，阻断污染数据 |
| **重复扫描** | 固定 7 天窗口，成本高、重复多、状态不可控 | 增量游标、调度记忆、动态回滚窗口 |
| **模型资源浪费** | 所有文章都调用旗舰模型，注意力浪费 | 轻量模型粗筛，旗舰模型深挖，分层调用 |
| **用户兴趣漂移** | 固定配置无法捕捉用户最近真实关切 | 短期焦点记忆 + 核心画像记忆 |
| **外部工具脆弱** | 抓取依赖外部脚本，失败后主系统无法理解原因 | 抓取能力内生化，运行时自检，异常恢复 |

这不是功能清单。这是我们踩过的坑，和每一个坑对应的结构性防御。

---

## 设计哲学

**1. 流程完成不等于任务成功。** 传统系统关注流程是否跑完。AI 系统必须关注判断是否可信。如果模型调用失败、结果置信度不足、分析链路中断，即使任务执行完成，也不能被视为成功。

**2. 宁可失败显性化，也不要错误自动流入下游。** AI 系统中最危险的错误不是报错，而是假成功。所以本系统拒绝静默降级：失败必须暴露，异常必须记录，污染数据必须阻断。

**3. 调度器应该带着记忆醒来。** 定时任务不应该每次都像第一次运行。它应该知道自己上次处理到哪里，哪些内容已经读过，哪些内容才是真正新增。

**4. 模型调用是一种认知预算。** 不是所有信息都值得旗舰模型处理。粗筛、精读、深度分析应该使用不同层级的模型能力。

**5. 用户兴趣不是配置项，而是持续演化的状态。** 用户真正关心什么，不只来自手动配置，也来自持续行为：重读、搜索、收藏、标记、追问。系统应该从这些行为中反推关注方向。

**6. Agent 的能力应该逐渐内生化。** 早期系统可以通过外部工具完成任务。但长期来看，关键技能应该逐步进入系统运行时。系统不应只会调用工具，还应知道工具是否可用、为什么失败、如何恢复。

## 系统架构

```
                  用户行为反馈
        搜索 / 重读 / 标记 / 订阅 / 忽略
                         ↓
                ┌────────────────┐
                │   记忆系统      │
                │ 短期焦点 / 核心画像 │
                └────────────────┘
                         ↓
RSS / 微信 / 网页源 → 增量调度器 → 内容抓取与解析
                         ↓
                  轻量模型初筛
                         ↓
                高价值候选内容池
                         ↓
                  旗舰模型深度分析
                         ↓
              结构化信号 / 观点 / 方法论
                         ↓
                知识库 / 推送 / 重读队列

异常链路：
LLM 失败 / 抓取失败 / 解析失败
        ↓
失败队列 + 告警 + 阻断污染数据
```

处理管线：

```
信息源 (URL / RSS / IM)
  ↓
QueueStore — 任务队列，SQLite 持久化
  ↓
RuntimeGuard — 运行时隔离，拒绝污染路径和幽灵进程
  ↓
Fetch — 多平台抓取，9+ 渠道自适应
  ↓
Score — 四层评分引擎
        L1 信号强度 × L2 来源可信度 × L3 内容质量 = 客观质量
        最终分 = 70% 客观质量 + 30% 个性化匹配 (L4)
  ↓
Extract — LLM 深度萃取，输出结构化知识卡片
  ↓
Output — Obsidian Markdown + Telegram 推送
```

评分合约：

```python
objective_quality = L1 * L2 * L3
final_score       = 0.70 * objective_quality + 0.30 * L4
score             = final_score * 10  # 0-10 分制
```

每一个环节的失败都有类型化处理——`retry_scheduled`、`rejected`、`failed_terminal`。绝不把失败标记为成功。

## 支持的平台

| 平台 | 抓取方式 | 说明 |
|------|---------|------|
| YouTube | yt-dlp | 视频元数据与字幕 |
| Twitter/X | xreach / MCP | 推文与线程 |
| Reddit | Jina Reader | 帖子与评论 |
| V2EX | Jina Reader | 讨论帖 |
| Hacker News | Jina Reader | 技术讨论 |
| 微信公众号 | Agent-Reach | 文章正文（内置技能） |
| 小宇宙 FM | Jina Reader | 播客元数据 |
| RSS/Atom | stdlib xml | 订阅源聚合 |
| Web | Jina Reader | 任意 URL |

**MCP 扩展：** 系统架构支持通过 MCP 协议接入外部信息源管理服务，实现搜索、频道管理和文章检索的扩展，无需修改核心代码。

**Web 搜索：** 支持 Exa API 直接调用或通过 MCP 模式工作，支持分类过滤、域名限制、时效筛选。

## 快速开始

### 安装

```bash
git clone https://github.com/bor799/information-tracking-agent.git
cd information-tracking-agent
pip install -e .
```

### 配置

```bash
cp config/config.example.yaml config/config.local.yaml
cp .env.example .env
```

编辑 `.env` 填入 API key（至少需要一个 LLM 提供商）。详细配置见 [docs/CONFIGURATION.md](docs/CONFIGURATION.md)。

### 运行

```bash
# 入队一条 URL
information-tracking-agent enqueue-url "https://example.com/article"

# 处理队列中所有待处理任务（staging 模式）
information-tracking-agent worker-once --limit 10

# 查看队列状态
scripts/control.sh status
```

## 项目结构

```
src/knowledge_extractor_v3/
  shadow_runner.py       # CLI 入口
  pipeline.py            # 处理管线编排
  worker.py              # 队列消费与批处理
  scheduler.py           # 增量源调度
  queue_store.py         # SQLite 任务队列（类型化状态机）
  runtime_guard.py       # 运行时隔离与指纹
  config_loader.py       # 分层配置加载
  models.py              # 数据模型
  prompt_registry.py     # Prompt 版本管理
  llm/                   # LLM 提供商抽象（多路由 + 熔断）
  fetchers/              # 多渠道抓取适配器（9+ 平台）
  sources/               # 信息源注册与去重
  outputs/               # Obsidian + Telegram 输出
```

40 个源文件，24 个测试文件，224+ 测试用例。

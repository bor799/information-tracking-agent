# V3 产品方向

目标：把 V3 从“脚本工具”调整为“后台知识服务 + 对话控制 + iOS 整理端”。

## 一句话定位

V3 负责持续抓取、萃取、入库；人通过对话发出意图；iOS 负责查看、整理、确认和复用。

## 交互变化

过去：

- 打开软件
- 点按钮
- 看日志
- 手动判断是否异常

现在应该变成：

- 你说一句话：检查一下 V3
- Agent 执行：预检、启动、恢复、汇总
- iOS 展示：今天抓到了什么、哪些失败、哪些值得读

## 产品分层

| 层 | 责任 |
| --- | --- |
| 后台服务 | 抓取、队列、LLM 分析、健康检查 |
| Agent 控制层 | 启动、恢复、诊断、总结 |
| iOS 整理端 | 收件箱、标签、摘要、稍后读、确认入库 |

## iOS 不该做什么

iOS 不负责跑 7x24H 后台任务，也不负责复杂抓取。iOS 是整理端，不是服务器。

## iOS 应该先做什么

MVP 只需要四个视图：

1. 今日新增
2. 待读/待整理
3. 失败与风险
4. 系统状态

## 技术路线

### Phase 1：先把后台跑稳

- `control.sh recover`
- heartbeat
- watchdog
- 队列重试

### Phase 2：暴露状态接口

- `/health`
- `/queue`
- `/items`
- `/actions/retry`

### Phase 3：做 iOS 轻客户端

- 只读列表
- 卡片详情
- 标记、归档、稍后读
- 远程触发 retry/recover

### Phase 4：对话成为主入口

典型指令：

```text
检查一下 V3 有没有在跑
把今天值得读的内容整理给我
失败的任务帮我重试
把这条内容归档到 AI 投资
```

## 判断标准

这个项目不是再多写几个脚本，而是把“信息处理”做成一个长期在线的软件系统。

好的结果是：你不监控系统，你只表达目标；系统自己运行、恢复、汇报。
